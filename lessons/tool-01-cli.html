The Command Line, Without Fear

Navigate, move files, and run programs from the terminal — the interface every data tool assumes you know.

18 min readBeginner-friendlyLesson 1 of 6 in track
◎ Why this matters

Every data scientist eventually meets a machine with no graphical interface — a cloud server, a training box, a colleague's Docker container. The command line is how you drive it. It's also the fastest way on Earth to answer "how many rows is this 4 GB file?" or "which values are in this column?" — without ever opening the file. The terminal isn't nostalgia; it's a data tool, and fluency here is one of the quiet signals that separates people who use data tools from people who build with them.

NavigationGetting around — the five verbs

A filesystem is a tree; the command line lets you walk it. Five commands cover 90% of movement:

terminal
pwd                 # print working directory — 'where am I?'
ls -lh              # list files, long form, human-readable sizes
cd projects/churn   # change directory (cd .. goes up one)
mkdir -p data/raw   # make folders (-p creates parents as needed)
cp a.csv data/      # copy;  mv renames/moves;  rm deletes (careful!)
Output
$ pwd
/home/you/projects/churn
$ ls -lh
-rw-r--r--  1 you  staff   4.2G  Jul 20 09:14  events.csv
-rw-r--r--  1 you  staff   2.1K  Jul 20 09:10  README.md
▲ Watch out

rm has no undo and no trash on the command line — a deleted file is gone. Never run rm -rf with a path you haven't triple-checked, and never with a variable you're not 100% sure of (rm -rf $DIR/ when $DIR is empty deletes /). This is the one place to slow down.

The data superpowerReading a file without opening it

This is where the terminal beats every spreadsheet. You can inspect a file too big to open in seconds — peek at the top, count the rows, find a value:

terminal
head -3 events.csv        # first 3 lines (see the header + shape)
tail -2 events.csv        # last 2 lines
wc -l events.csv          # count lines (rows)
grep "signup" events.csv  # every line containing 'signup'
cut -d, -f2 events.csv    # pull column 2 (comma-delimited)
Output
$ head -3 events.csv
user_id,event,ts
1042,signup,2026-07-19T10:02
1042,purchase,2026-07-19T10:31

$ wc -l events.csv
 8123456 events.csv

The Unix philosophyPipes — small tools, chained

The magic is the pipe (|): it feeds one command's output into the next. Each tool does one thing; you compose them into an answer. The classic "count the distinct values in a column, most common first" is a one-liner — no Python, no pandas:

terminal
cut -d, -f2 events.csv | sort | uniq -c | sort -rn | head
#  ^ column 2         ^sort  ^count   ^by count  ^top
Output
$ cut -d, -f2 events.csv | sort | uniq -c | sort -rn | head
 4120233 view
 2891004 click
  812440 purchase
  299779 signup
✦ Mentor's tip

Read a pipeline left to right as a sentence: take column 2, sort it, count runs of each value, sort those counts descending, show the top. You just did a GROUP BY ... COUNT ORDER BY on a multi-gigabyte file with five tiny programs and no memory blow-up — because each tool streams its input.

Interactive labTry the thinking — a pipeline in Python

◆ LABLab — 'uniq -c | sort -rn' in Python

The CLI pattern cut | sort | uniq -c | sort -rn | head -1 finds the most common value. Reproduce its logic in Python: given the events list of rows, count how often each event appears and assign the most common event name (a string) to answer.

events (list of (user_id, event) tuples) is preloaded. First Run may take a moment to boot Python.
Your result
Hint. collections.Counter with .most_common(1) mirrors uniq -c | sort -rn | head -1. most_common(1) returns [(name, count)] — take [0][0] for the name.
Reference solution.
from collections import Counter
counts = Counter(ev for _uid, ev in events)
answer = counts.most_common(1)[0][0]
view appears most (4 times). Counter is the pipeline — the same count-then-rank idea, whether you express it in five shell tools or one Python line.


★ Key points to remember
  • The terminal is how you drive headless machines (servers, containers) — unavoidable in real DS work.
  • Navigate with pwd / ls / cd / mkdir / cp · mv · rm. rm has no undo — slow down.
  • Inspect huge files instantly: head / tail / wc -l / grep / cut — no need to open them.
  • The pipe | chains one-job tools into an answer; each tool streams, so size doesn't matter.
  • cut | sort | uniq -c | sort -rn is a GROUP-BY-COUNT — think in flows.

◆ Practice▸
1In one pipeline, print the 5 most common values of the 3rd (comma-separated) column of log.csv, each with its count.
Show solution & reasoning
cut -d, -f3 log.csv | sort | uniq -c | sort -rn | head -5. Read it left to right: pull column 3, sort so identical values are adjacent, uniq -c counts each run, sort -rn orders by count descending (numeric), head -5 keeps the top five. (Add tail -n +2 after cut if you want to drop the header row first.)
2Why is chaining small tools with pipes often better than writing one big script?
Show solution & reasoning
Each tool does one job well and streams its input, so a pipeline uses almost no memory even on huge files, is easy to build up and inspect one stage at a time, and reuses battle-tested programs instead of new, buggy code. It's the Unix philosophy: compose simple, sharp tools rather than build monoliths. (When logic gets complex or needs tests, then reach for Python.)


◎ Deep dive: grep/awk/sed, working on remote servers, and what streams vs. what doesn't▸
Beyond the basics, three tools pay for themselves. grep (with -i case-insensitive, -v invert, -c count, -E regex) finds lines; awk does per-column logic (awk -F, '$3>100' keeps rows where column 3 exceeds 100 — a filter with no pandas); and sed does stream find/replace (sed 's/,/\t/g' turns a CSV into a TSV). Together they cover a startling share of "quick data question" tasks.
Working on a remote machine. ssh user@host opens a shell on a server; scp file user@host:path copies files to/from it; and redirects (> out.txt writes, >> out.txt appends, 2>&1 folds errors into output) let you capture results. A long job survives a dropped connection if you run it inside tmux or with nohup ... &. These four ideas — ssh, scp, redirects, tmux — are the daily reality of training models on hardware that isn't your laptop.
A caution on giant files. Streaming tools (wc, grep, cut, awk) scale to any size because they never hold the whole file in memory. But sort does buffer (it may spill to disk), and anything that loads the file at once (an editor, pandas.read_csv without chunksize) will struggle. Knowing which operations stream and which don't is what lets you work confidently with data far bigger than RAM.
