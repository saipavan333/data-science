◆ Track 8 · Classical Machine Learning

Track 8 Interview Check

Core ML questions and case prompts.

15 min readBeginner-friendlyLesson 8 of 9 in track
◎ Why this matters

The machine-learning interview round rarely asks you to code an algorithm from scratch. It tests something harder to fake: do you understand the tradeoffs, can you tell when a model is lying to you, and could you build one without fooling yourself? This lesson is the ML question bank — the recurring prompts with answers you can say out loud, plus one runnable reality-check that demolishes the most seductive trap in the field.

Interview patternThe one that trips everyone: "99% accurate — are you happy?"

This is the most-asked ML judgement question, and the answer is almost always "not yet — it depends on the base rates." If 90% of cases are one class, a model that blindly predicts that class is 90% accurate and completely useless. Always compare against a baseline. Watch it: here the data is imbalanced (10% positive), so a do-nothing baseline already scores 0.90 — the real model has to beat that, not zero:

baseline_check.py
from sklearn.datasets import make_classification
from sklearn.model_selection import train_test_split, cross_val_score
from sklearn.dummy import DummyClassifier
from sklearn.ensemble import RandomForestClassifier

# imbalanced: only ~10% of cases are the positive class
X, y = make_classification(n_samples=600, weights=[0.9, 0.1], n_features=8, n_informative=5, random_state=4)
Xtr, Xte, ytr, yte = train_test_split(X, y, stratify=y, test_size=0.25, random_state=4)

baseline = DummyClassifier(strategy="most_frequent").fit(Xtr, ytr)
print(f"baseline (always predict the majority class): {baseline.score(Xte, yte):.2f} accuracy")

rf = RandomForestClassifier(n_estimators=200, random_state=0)
print(f"random forest — 5-fold CV on train:          {cross_val_score(rf, Xtr, ytr, cv=5).mean():.2f}")
rf.fit(Xtr, ytr)
print(f"random forest — held-out test:               {rf.score(Xte, yte):.2f}")
Output
baseline (always predict the majority class): 0.89 accuracy
random forest — 5-fold CV on train:          0.95
random forest — held-out test:               0.97

The lazy baseline already hits 0.90 just by never predicting the rare class — so "90% accurate" would be worthless here, and even the forest's accuracy has to be read against that floor. On imbalanced data you stop quoting accuracy and switch to precision, recall, F1, or ROC-AUC (Track 10). Always beat, and report against, a baseline.

Interview patternExplain the bias–variance tradeoff

Say it as a story. Bias is error from a model too simple to capture the real pattern (it underfits — a straight line through a curve). Variance is error from a model so flexible it fits the noise (it overfits — changes wildly with the data). Total error is the sum, and they trade off: reduce one and you tend to raise the other. The goal is the sweet spot in between. Then connect it to models: a single deep tree is high-variance; bagging (random forests) cuts that variance by averaging; boosting starts from high-bias shallow trees and cuts bias by sequential correction; regularization trades a little bias for less variance. Naming those levers shows you can act on the tradeoff, not just recite it.

Interview pattern · the walk-through"How would you build and validate a model?"

They want a process, said calmly in order:

  1. Frame it — regression or classification? What's the target, and what's the cost of each kind of mistake?
  2. Split first — hold out a test set immediately (stratified or by time), and don't touch it.
  3. Baseline — a dummy/majority or simple model, so you know what "good" even means.
  4. Build in a Pipeline — preprocessing + model together, so scaling/imputing is fit per fold and can't leak.
  5. Cross-validate to compare models and tune hyperparameters — never on the test set.
  6. Open the test set once for a final, honest estimate.
  7. Interpret & sanity-check — which features drive it, does it beat the baseline, any leakage, does the error pattern make sense?

That sequence — frame, split, baseline, pipeline, CV/tune, test-once, interpret — is the whole of applied ML, and reciting it in order signals you've actually shipped.

★ Key points to remember
  • Accuracy alone can lie — on imbalanced data a do-nothing baseline scores high; always compare against it, and use precision/recall/AUC when classes are skewed.
  • Bias–variance: too simple = underfit (high bias); too flexible = overfit (high variance); forests cut variance (bagging), boosting cuts bias, regularization trades between them.
  • Trees need no scaling; distance/gradient models (k-means, logistic, SVMs) do.
  • The build process: frame → split → baseline → pipeline → CV/tune → test once → interpret.
  • Guard against leakage (fit preprocessing inside CV) and never tune or select on the test set.

◆ Practice▸
1An interviewer asks: "You have 50,000 rows and want to predict churn. Walk me through your approach." Give a crisp answer.
Show solution & reasoning
"First I'd frame it — binary classification, and the cost of a false negative (missed churner) vs false positive (wasted retention offer) drives my metric, likely recall or a cost-weighted score. I'd split out a stratified test set immediately and lock it away, check for leakage (drop features that only exist after churn). I'd set a baseline (majority class), build a Pipeline (preprocessing + model) so nothing leaks, and use cross-validation to compare a logistic-regression baseline against a gradient-boosted model and to tune it. Then I'd open the test set once for a final estimate, and interpret — feature importances, does it beat the baseline, does the error make business sense." That order is the answer.
2When would you prefer logistic regression over gradient boosting, even though boosting is usually more accurate?
Show solution & reasoning
When interpretability and trust matter more than the last few points of accuracy: regulated settings (credit, insurance, healthcare) where you must explain each decision, where every coefficient is an auditable odds ratio; when you have little data (boosting overfits small sets); when you need a fast, cheap baseline or low-latency scoring; or when the relationship really is roughly linear in the log-odds. "Most accurate" isn't the only axis — explainability, data size, latency, and maintenance all count.


◆ Say these out loud before your ML interview
Before moving on, make sure you could answer these out loud:
  • Explain the bias–variance tradeoff, and name a lever for each side.
  • Bagging vs boosting — what does each reduce, and how?
  • "The model is 99% accurate" — what do you ask before believing it?
  • How do you prevent overfitting? (limit flexibility, more data, regularization, CV, early stopping)
  • What is data leakage, and how does a Pipeline prevent it?
  • Walk me through building and validating a model end to end.
  • When would you choose an interpretable model over a more accurate one?


◆ Interview-ready
The through-line of this whole track: models are easy to run and easy to fool yourself with. The senior signal is judgement — comparing to a baseline, reading the bias–variance tradeoff, refusing to trust accuracy on imbalanced data, and validating without leakage. Show that and the coding is a formality.
