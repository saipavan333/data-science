◆ Track 3 · The Data Scientist's Toolkit

Notebooks vs Scripts & Project Structure

When to explore in a notebook, when to write a script, and how to lay a project out.

16 min readBeginner-friendlyLesson 4 of 7 in track
◎ Why this matters

The Jupyter notebook is where data science thinks out loud: run a cell, see a chart, adjust, run again — a conversation with your data. It's the right tool for exploration, teaching, and sharing analysis. But its greatest strength — running cells in any order, keeping everything in memory — is also a trap that produces results nobody (including you) can reproduce. This lesson is how to get the good without the bad.

Cells, kernel, stateWhat a notebook actually is

A notebook is a list of cells (code or Markdown prose) attached to a kernel — a live Python process that remembers everything you've run. Run a code cell and its variables persist in the kernel's memory, available to every later cell. That persistence is what makes exploration fluid: load a big dataset once, then poke at it across many cells without reloading.

notebook
# Cell 1 — load once, stays in memory
import pandas as pd
df = pd.read_csv('events.csv')

# Cell 2 — explore, re-run freely without reloading
df['event'].value_counts().head()
Output
view        4120233
click       2891004
purchase     812440
signup       299779
Name: event, dtype: int64
✦ Mentor's tip

Two productivity multipliers: Markdown cells turn a notebook into a narrated report (headings, explanation, conclusions between the code), and magics — %timeit some_func() to benchmark a line, %matplotlib inline for charts, !pip install X or !ls to run shell commands — give you a lab bench inside the page.

The #1 notebook bugThe hidden-state trap

✗ Common pitfall

Because the kernel remembers everything, a notebook can look correct while being impossible to reproduce. Two classic ways it lies:

  • Out-of-order execution. You run cell 5, then edit and run cell 2, then cell 8. The numbers on screen reflect a history no one can retrace — the execution counts (In [17]) jump around. A fresh run top-to-bottom may give different results, or crash.
  • Zombie variables. You define df in a cell, later delete that cell, but df still lives in the kernel — so later cells keep working until you restart, when they suddenly fail with NameError. The notebook depended on code that no longer exists.

Both mean the notebook you see isn't the notebook that will run.

The cure is a habit: before trusting or sharing any result, Kernel → Restart & Run All. This wipes memory and runs every cell top-to-bottom in order — exactly how someone else will run it. If it completes cleanly and reproduces your numbers, the notebook is honest. If it breaks, you just caught a hidden-state bug before it embarrassed you in a review. Do this religiously.

Right tool, right jobWhen NOT to use a notebook

Notebooks are for exploration and narrative. They're the wrong home for code that needs to be reused, tested, or deployed — shared functions, data pipelines, production model code. That belongs in .py modules you can import, unit-test, and version cleanly (notebook JSON makes ugly, unreviewable Git diffs). The mature pattern: prototype in a notebook, then graduate the keep-worthy logic into .py files that the notebook — and production — both import.

◎ Why this matters

A good tell of a senior data scientist: their notebooks are clean narratives that call into tested library code, not 400-cell monsters where all the logic lives inline. The notebook shows the story; the .py modules hold the engine.

★ Key points to remember

  • A notebook = cells + a kernel (a live process that remembers every variable you've run) — great for exploration.
  • Markdown cells narrate; magics (%timeit, !shell) add power inside the page.
  • The hidden-state trap: out-of-order runs and zombie variables make results that can't be reproduced.
  • Always Restart & Run All before trusting or sharing — it runs top-to-bottom the way others will.
  • Explore in notebooks; move reusable/production code into tested .py modules the notebook imports.

◆ Practice▸
1Give three concrete habits that keep notebooks reproducible.
Show solution & reasoning

(1) Restart & Run All before trusting or sharing any result — it runs cells top-to-bottom on a clean kernel, the way others will. (2) Keep cells in logical top-to-bottom order and avoid editing-and-rerunning earlier cells after later ones depend on them; if you must, restart. (3) Move reusable logic into imported .py modules and pin your environment (requirements) so the notebook's dependencies are explicit. Bonus: clear outputs before committing, or use a tool like nbstripout, so Git diffs stay readable.

2Why do notebooks produce painful Git diffs, and what can you do about it?
Show solution & reasoning

A .ipynb is JSON that stores code, outputs, execution counts, and image data inline, so even a one-character code change produces a huge, unreadable diff (and merge conflicts in output blobs). Mitigations: strip outputs before committing (nbstripout or Kernel → Clear Outputs), keep heavy/reusable code in .py files that diff cleanly, and for review, share the rendered notebook (nbviewer/HTML) while versioning the logic as modules. Tools like Jupytext can even pair a notebook with a plain-.py representation for sane diffs.

◎ Deep dive: nbconvert/papermill/Jupytext, VS Code cells, and prototype-to-production▸

Notebooks in a serious workflow. The ecosystem has grown up: nbconvert turns a notebook into HTML/PDF/slides for sharing; papermill parameterises and executes notebooks programmatically (run the same analysis for 50 clients by injecting parameters), which is how notebooks earn a place in scheduled pipelines; and Jupytext pairs each .ipynb with a plain .py twin so version control sees clean text. VS Code and modern IDEs also run notebook cells against .py files directly (# %% cell markers), giving you the exploratory feel with diff-friendly source.

The prototype-to-production arc. A healthy project life-cycle: explore in a notebook → as logic stabilises, extract functions into src/ modules with tests → the notebook becomes a thin narrative that imports those modules → production schedules the module code (not the notebook). Nothing about this throws away the notebook — it stays as living documentation of how the analysis was reasoned out. The skill is knowing when a cell has earned promotion from scratchpad to tested code.
