◆ Track 10 · Model Evaluation & Interpretation

Validation Done Right

Splits that don't lie — including time and groups.

18 min readBeginner-friendlyLesson 2 of 6 in track
◎ Why this matters

A model's score on the data it trained on is worthless — of course it does well on questions it has already seen the answers to. The entire point of evaluation is to estimate how the model will do on data it has never seen. Getting that estimate honest — and keeping it honest — is validation strategy, and it's where a huge fraction of real-world "our model was great in testing and failed in production" disasters are actually born.

The sacred splitThree roles: train, validate, test

Split your data into three jobs:

  • Training set — the model learns its parameters here.
  • Validation set — you compare models and tune hyperparameters here, choosing what works best.
  • Test set — a locked vault opened once, at the very end, for a single honest estimate of real-world performance.

The golden rule: the test set is sacred. Every time you look at it and adjust anything, you leak a little of it into your choices, and its estimate becomes optimistic. Touch it once.

✗ Common pitfall

Tuning against your test set turns it into a training set. If you try 40 models and pick the one with the best test score, that score is no longer an unbiased estimate — you've overfit to the test set by selection. That's exactly why the validation set exists (to absorb all your experimentation) and why the test set is opened only for the final, already-decided model.

k-foldCross-validation: a better estimate from limited data

A single train/validation split is wasteful (the validation rows never train the model) and noisy (you got lucky or unlucky with which rows landed in validation). k-fold cross-validation fixes both: split the data into k parts, then train k times, each time holding out a different part for validation. Average the k scores for a far more stable estimate, and every row gets used for both training and validation:

cross_val.py
import numpy as np
from sklearn.model_selection import cross_val_score
from sklearn.linear_model import LogisticRegression
from sklearn.datasets import make_classification

X, y = make_classification(n_samples=600, n_features=8, random_state=0)
scores = cross_val_score(LogisticRegression(max_iter=1000), X, y, cv=5)

print("per-fold accuracy:", np.round(scores, 3))
print(f"honest estimate:   {scores.mean():.3f}  ±  {scores.std():.3f}")
print("(report the mean AND the spread — one split could have landed anywhere in that range)")
Output
per-fold accuracy: [0.942 0.975 0.942 0.975 0.917]
honest estimate:   0.950  ±  0.022
(report the mean AND the spread — one split could have landed anywhere in that range)

Stratified, grouped, time-seriesPick the CV that matches your data

Plain k-fold assumes rows are interchangeable. Often they're not — match the scheme to the structure or you'll leak:

  • Stratified k-fold — keeps each fold's class balance equal to the whole. Essential for imbalanced classification (else a fold might contain almost no positives).
  • Grouped k-fold — keeps all rows from one entity (a patient, a user, a store) together in the same fold. Without it, the model sees the same patient in train and validation — leakage that inflates scores.
  • Time-series CV — always train on the past, validate on the future (expanding/rolling windows). A random split on temporal data lets the model peek at the future — invalid.
▲ Watch out

The most common validation leak is a random split on data that has groups or time. If the same user, customer, or patient can appear in both train and test, or if you shuffle time-ordered rows, your estimate is optimistic and will not survive production. Ask "what's the unit that must not straddle the split?" before choosing a CV scheme.

★ Key points to remember
  • Split into train (learn), validation (tune/compare), and test (one final honest estimate).
  • The test set is sacred — touch it once. Tuning against it turns it into a training set and inflates the score.
  • k-fold CV gives a more stable, data-efficient estimate than one split — report mean ± spread.
  • Match the scheme: stratified (imbalance), grouped (repeated entities), time-series (temporal) — or you leak.
  • The classic leak: a random split on grouped or time-ordered data — ask what unit must not straddle the split.

◆ Practice▸
1Explain why cross-validation is usually preferable to a single train/validation split, and one case where you might NOT use plain k-fold.
Show solution & reasoning
Why CV is better: a single split wastes the validation rows (they never train the model) and is noisy — your estimate swings depending on which rows happened to land in validation. k-fold trains k times on different splits and averages, giving a more stable, lower-variance estimate that uses every row for both roles, plus a spread across folds that tells you how uncertain the estimate is. When not to use plain k-fold: when rows aren't exchangeable — time-ordered data (use forward-chaining time-series CV so you never train on the future), grouped data (use grouped k-fold so an entity doesn't straddle folds), or severe imbalance (use stratified k-fold so each fold keeps the class ratio). Plain k-fold silently leaks or destabilises in all three.
2What is nested cross-validation, and what problem does it solve?
Show solution & reasoning
Nested CV runs two loops: an inner CV that selects hyperparameters, wrapped inside an outer CV that estimates performance. For each outer fold, you do a full inner-CV tuning on that fold's training portion, then score the chosen model on the outer fold's held-out data — which the tuning never touched. Problem it solves: if you tune and evaluate on the same CV, the reported score is optimistic because hyperparameters were chosen using the very data you're scoring on (selection leakage). Nested CV keeps model selection and performance estimation on separate data, yielding an unbiased estimate of the whole modelling-plus-tuning pipeline — at higher compute cost, which is why it's used when an honest estimate really matters (small data, research, high stakes).


◎ Deep dive: choosing fold count, evaluation variance, and overfitting the validation set▸
How big should each split be, and how many folds? With plenty of data, a simple 60/20/20 train/val/test split is fine and cheap. As data shrinks, cross-validation earns its keep: 5- or 10-fold are the standard compromises (more folds → less bias, more variance and compute; leave-one-out is the extreme, often high-variance and rarely worth it). The deeper point is that evaluation itself has variance — report the spread across folds, and when comparing two models, ask whether their CV-score difference is larger than that fold-to-fold noise before declaring a winner.
The subtle enemy: overfitting to the validation set. Even with a proper split, running dozens of experiments and always keeping the best validation score means you're slowly fitting to the validation set through selection — the 'garden of forking paths' from the experimentation track, wearing a modelling hat. Defenses: limit how much you sculpt to validation, use nested CV for an honest outer estimate, and above all keep the test set truly untouched until the end. A useful mental model: the test set is a one-shot exam — the moment you use it to make a decision, you've spent it, and any further use only tells you how well you've memorised the exam, not how well you'll do on new questions.
